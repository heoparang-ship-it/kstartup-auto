# 신혼생활 공고 매칭 기준 (SSOT) — v5

> 이 문서는 founder-gov-radar 스킬이 공고 티어를 판정하는 **유일한 기준 문서**입니다.
> 기준을 바꾸려면 이 파일만 수정하세요. 다음 실행 시 전체 풀이 자동 재분류됩니다.

> **v5 변경점 (2026-04-22)** — classify.py **v9.2** 짝 · **"스토리텔링 브리지 게이트웨이"** 도입
>
> 1. **🔥 CORE PIVOT — 도메인 화이트리스트 필수 요건 폐지.** 기존 v4는 "L1~L7 레이어 or L0 범용 키워드 0개 → 무조건 RED"라는 하드 필터를 걸었는데, 이로 인해 실제로는 **신혼부부 스토리텔링으로 지원 가능한 공고가 대량 RED 처리**되고 있었다(2026-04-22 시점 160/328건 = 49%가 이 이유로 차단). **스마트미디어콘텐츠지원사업처럼 도메인 표면은 관련 없어 보이지만 "신혼부부 생애주기 플랫폼"으로 스토리텔링을 풀어내면 충분히 선정되는 공고**를 놓쳐왔다. v5는 이 필터를 스토리텔링 브리지 게이트웨이로 교체.
> 2. **DOMAIN_NEGATIVE_HARD 축소** — 9카테고리 → **3카테고리로 축소** (국방·군수 / 바이오·신약 / 농수축임). 나머지는 물리적으로 지원 불가한 수준이 아니라 단순히 "도메인 표면 거리가 멀 뿐"이라 브리지 가능. soft 플래그로 이동.
> 3. **도메인 네거티브 SOFT 신설** — 도시안전·치안, 제조공정, 의료기기 전시·인증, 로봇, 건설토목, 에너지발전, 관광B2B, TRL증빙, 양자컴퓨팅, 우주항공 — **1 hit라도 있으면 ORANGE 강등** (RED 아님). 스토리텔링 브리지로 역전 가능하기 때문.
> 4. **스토리텔링 브리지 키워드 확장** — L0 범용 플랫폼(11개) → **L0 범용 플랫폼 + BRIDGE 32개**로 확장. 창업·스타트업·벤처·청년·액셀러레이팅·콘텐츠·크리에이터·여성·소상공인·1인 창조기업·테크·디지털 등. 이 키워드 1+ hit면 최소 ORANGE 보장.
> 5. **스토리텔링 브리지 게이트웨이 (신규)** — 라이프사이클 0개 + L0/BRIDGE 0개 + 도메인 네거티브 하드 0개인 경우에만 최종 RED. 즉 **브리지 가능성이 아예 없는 공고만 RED**.
> 6. **티어 로직 재조정** — GREEN(1순위)=강 매칭+지역OK / YELLOW(2순위)=라이프사이클 1+ or 플랫폼 직접 매칭 / ORANGE(3순위)=스토리텔링 브리지 후보 / RED=자격·지역·도메인 완전 불가.
> 7. **"× 마감" 라벨 분리** — 대시보드는 `expired_reds`(실제 마감일 경과)와 `filter_reds`(도메인/자격 미스매치)를 분리 표기. 기존 통합 "× 마감" 표기가 오해를 유발(실제 마감 ~20건 / 필터 제외 ~300건을 구분 안 함).

## 🌉 스토리텔링 브리지 게이트웨이 (v5 신규)

> 사용자 직접 지시 (2026-04-22): **"실제로 할만한게 많았다고 그전에. 로직 변경해야해. 예를들어 지금선정된 스마트미디어콘텐츠도 우리 사업이랑 관련없지만 신혼부부로 스토리텔링을 풀어내서 선정된거라고. 우리걸로 풀어내기만 하면 돼."**

### 핵심 원칙

**공고의 도메인 표면이 직접 우리 사업(신혼생활)과 일치하지 않아도, "신혼부부 생애주기 플랫폼"이라는 당사자성 서사로 풀어낼 수 있으면 지원 가능하다.**

선정 사례 검증:
- 스마트미디어콘텐츠지원사업 → "신혼부부 미디어커머스" 서사로 선정 (도메인 표면 ≠ 신혼)
- 로컬크리에이터 → "인천 신혼 라이프스타일 콘텐츠" 서사로 선정
- 청년창업지원 → "신혼부부 당사자 창업" 서사로 선정
- K-Startup 챌린지 → AI 기반 정책 매칭 서비스로 포지셔닝

### 브리지 가능성 3단계

| 단계 | 조건 | 기대 tier |
|---|---|---|
| **직접 매칭** | 라이프사이클 L1~L7 키워드 2+ hit | GREEN |
| **직접 매칭 부분** | 라이프사이클 L1~L7 키워드 1 hit OR 플랫폼 L0 직접 hit | YELLOW |
| **스토리텔링 브리지** | 라이프사이클 0 + BRIDGE 키워드 1+ hit | ORANGE |
| **브리지 불가** | 라이프사이클 0 + BRIDGE 0 + DOMAIN_NEGATIVE_HARD 미해당 | RED (정보 부족) |

| 엔티티 | 지역 | 단계 | 자격 키워드 (매칭 시 green) | 실격 키워드 (매칭 시 red) |

| **E1 허파랑 개인** | 인천 | 예비창업자 | 예비창업, 개인, 1인 창조기업, 전국, 인천 | 법인 필수, 사업자등록 필수, 5년 이상 업력 |
| **E2 XCom 법인** | 인천 | 초기창업(업력 ~6개월 · 2025-10-10 설립) | 초기창업, 창업 3년 이내, 창업 7년 이내, 법인, 인천, 소상공인 | 예비창업 한정, 7년 초과, 대기업 |

지역 일반 규칙:
- **E1·E2 (인천)**: region ∈ {전국, 인천, 수도권} 또는 agency에 인천 기관 → green. 비수도권 단독 → red. 서울 단독 → red.
- **E3 (대구)**: region ∈ {전국, 대구, 경북, 영남} 또는 agency에 대구·경북 기관 → green. 인천 단독/서울 단독/기타 지역 → red.

## 티어 결정 로직 (v5)

```
# 0. 만료 체크
deadline < TODAY → RED_expired

# 1. 도메인 네거티브 HARD 체크 (축소된 3 카테고리)
DOMAIN_NEGATIVE_HARD 1+ hit → 무조건 RED (엔티티 무관)

# 2. 도메인 매칭 스캔
lifecycle_hits = scan_lifecycle(text)       # L1~L7
l0_hits        = scan_l0(text)              # 플랫폼 직접
bridge_hits    = scan_bridge(text)          # BRIDGE 32종
soft_neg_hits  = scan_negative_soft(text)   # SOFT 10 카테고리

# 3. 브리지 게이트웨이 (v5 핵심)
if lifecycle_hits == 0 and l0_hits == 0 and bridge_hits == 0:
    → RED (정보 부족: 브리지 가능성 0)

# 4. 엔티티 3종 독립 평가 (지역·단계)
for each entity:
    if region == red or stage == red:
        entity_tier = red
        continue
    # tier 부여
    if 강 매칭 트리거 hit and region green and stage ∈ {green, yellow}:
        entity_tier = green
    elif lifecycle ≥ 2 and region green and stage green:
        entity_tier = green
    elif lifecycle == 1 and region green:
        entity_tier = yellow
    elif l0 ≥ 1 and region green:
        entity_tier = yellow
    elif bridge ≥ 1 and region green:  # 🌉 BRIDGE 경로
        entity_tier = orange
    elif region yellow and (lifecycle or l0 or bridge):
        entity_tier = orange
    else:
        entity_tier = red

    # green 감사 강등 (기존 v4 로직 유지)
    if entity_tier == green and audit_flags:
        entity_tier = yellow

    # SOFT 네거티브 hit 시 한 단계 강등
    if soft_neg_hits and entity_tier in {green, yellow}:
        entity_tier = {green→yellow, yellow→orange}[entity_tier]

# 5. best_entity / best_tier
best = max(entities, tier_rank)

# 6. nogo 피드백 루프 (1단계 하향, 기존 유지)
if nogo_match and best_tier ∈ {green, yellow}:
    best_tier = demote(best_tier)
```

## Sonnet Verdict 프롬프트 (deep verdict용)

🟢 전체 + 🟡 상위 10건 + 🟠 상위 10건(신규)은 Sonnet이 공고 원문/PDF를 읽고 답변:

1. **매칭 경로**: L1~L7 직접 매칭 / L0 플랫폼 직접 / 🌉 스토리텔링 브리지 / 브리지 불가
2. **브리지 전략** (브리지 경로인 경우): 이 공고를 신혼생활 서사로 어떻게 풀어낼 것인가? 2줄 이내.
3. **도메인 네거티브 체크**: HARD 3종 해당? SOFT 10종 해당? 해당 시 tier 조정 근거.
4. **엔티티별 자격**: E1·E2·E3 각각 자격 요건 충족하는가?
5. **추천 엔티티**: 지원 시 가장 유리한 엔티티는? 1줄 이유.

출력 스키마 (v5):

## 용어 일치

- "E1/E2/E3" = 엔티티 3종 코드 (허파랑 개인 / XCom 법인 / 이영림 대구)
- "L1~L7" = 생애주기 레이어 (결혼준비·허니문·콘텐츠·이벤트·육아·주거·금융)

- "BRIDGE" = 스토리텔링 브리지 키워드 (v5 신규, ORANGE 보장)
- "domain.whitelisted" = 라이프사이클 or L0 레이어 1개 이상 hit
- "domain.bridgeable" = BRIDGE 1+ hit (v5 신규)
- "best_entity / best_tier" = 3개 엔티티 중 최고 tier 엔티티
- "audit_flags" = green → yellow 강등 사유 배열
- "negative_hard" = DOMAIN_NEGATIVE_HARD 히트 (3 카테고리)
- "negative_soft" = DOMAIN_NEGATIVE_SOFT 히트 (10 카테고리, tier 한 단계 하향)
- "nogo_history" = 사용자 NO-GO 누적 기록
